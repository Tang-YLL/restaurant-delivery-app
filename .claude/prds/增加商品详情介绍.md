# PRD: 增加商品详情介绍

## Executive Summary

本功能旨在增强商品详情页的内容展示能力，通过添加富文本格式的商品详细介绍，包括制作工艺、食材来源、营养价值、食用建议等丰富信息，提升用户购买决策体验和信任度。

**核心价值**：
- 提供更全面的商品信息，降低用户决策门槛
- 增强商品吸引力，提升转化率
- 建立品牌信任感，提高用户粘性
- 为后续营销内容奠定基础

---

## Problem Statement

### 当前问题

1. **信息不够详细**：现有商品详情页仅显示简短的description（50-100字），无法满足用户对商品信息的深度需求

2. **展示形式单一**：纯文本展示，缺乏视觉吸引力，无法突出商品特色和卖点

3. **用户决策困难**：缺少关键信息（如营养成分、适用人群、食用方法），用户难以快速做出购买决策

4. **营销能力有限**：无法通过故事化内容、品牌理念等软性内容提升商品价值感

### 为什么现在需要

- **竞争压力**：同类外卖平台已提供丰富的商品详情内容
- **用户期望提升**：用户对商品透明度的要求越来越高
- **差异化竞争**：通过优质的商品内容建立平台差异化优势
- **转化率优化**：详细的内容可以显著提升购买转化率

## User Stories

### 主要用户画像

**画像1：健康意识强的用户**
- 年龄：25-40岁
- 特征：关注营养成分、食材来源、热量等信息
- 痛点：现有信息无法满足其对健康的关注

**画像2：美食爱好者**
- 年龄：20-45岁
- 特征：对制作工艺、菜品文化、口味特点感兴趣
- 痛点：缺少深入了解菜品特色的渠道

**画像3：特殊饮食需求用户**
- 年龄：全年龄段
- 特征：有过敏源、宗教信仰、减肥等特殊需求
- 痛点：无法快速判断商品是否适合自己

## Requirements

### Functional Requirements

#### FR1: 商品详情内容展示
**需求ID**: FR-001
**优先级**: P0（必须）

- 显示商品详细介绍的独立分区
- 支持以下内容类型：
  - 菜品故事/品牌故事
  - 制作工艺说明
  - 食材来源介绍
  - 营养价值说明
  - 食用方法/建议
  - 适用场景/人群

#### FR2: 富文本编辑器（管理后台）
**需求ID**: FR-002
**优先级**: P0（必须）

- 管理后台支持富文本编辑商品详情
- 提供以下编辑功能：
  - 文本格式化（标题、段落、加粗、斜体、颜色）
  - 插入图片
  - 插入视频链接
  - 插入表格
  - 插入列表（有序/无序）
  - 撤销/重做

#### FR3: 营养成分展示
**需求ID**: FR-003
**优先级**: P1（重要）

- 以表格形式展示营养成分
- 必须字段：热量、蛋白质、脂肪、碳水化合物、钠
- 可选字段：膳食纤维、糖、胆固醇、维生素等
- 单位：每100克和每份
- 参考值百分比（NRV%）

#### FR4: 食材和配料信息
**需求ID**: FR-004
**优先级**: P1（重要）

- 显示主要食材列表
- 标注食材来源（如：本地采购、有机食材）
- 过敏源提示（如：含花生、蛋、奶制品）
- 食材新鲜度说明

#### FR5: 制作工艺展示
**需求ID**: FR-005
**优先级**: P2（次要）

- 图文结合展示制作步骤
- 支持多图轮播
- 突出特色工艺和厨师故事
- 预计制作时间说明

#### FR6: 移动端优化
**需求ID**: FR-006
**优先级**: P0（必须）

- 响应式布局，适配不同屏幕尺寸
- 图片懒加载和优化
- 流畅的滚动体验
- 快速加载（2秒内完成首屏渲染）

#### FR7: 内容版本控制
**需求ID**: FR-007
**优先级**: P2（次要）

- 支持内容草稿保存
- 支持内容历史版本查看
- 支持定时发布

#### FR8: 商品推荐算法
**需求ID**: FR-008
**优先级**: P1（重要）

- 基于分类的相似商品推荐
- 基于销量的热销推荐
- 基于用户行为的个性化推荐（后期）

### Non-Functional Requirements

#### NFR1: 性能要求
- 商品详情页首屏加载时间 < 2秒
- 富文本内容渲染时间 < 1秒
- 支持1000+并发访问
- 图片资源自动CDN加速

#### NFR2: 安全要求
- 富文本内容必须经过XSS过滤
- 图片上传必须进行格式和大小校验
- 敏感词过滤
- 操作日志记录

#### NFR3: 可扩展性
- 支持多语言扩展（中英文）
- 支持内容模板化
- 预留视频展示接口
- 预留AR/VR展示接口

#### NFR4: 可维护性
- 前后端分离，便于独立迭代
- 内容与样式分离
- 提供内容审核机制
- 完善的错误处理和日志

## Constraints & Assumptions

### 技术约束

1. **前端技术栈**：Flutter（移动端）、Vue3（管理后台）
2. **后端技术栈**：FastAPI + SQLAlchemy
3. **数据库**：SQLite（开发）、支持迁移到PostgreSQL
4. **富文本引擎**：Quill.js 或 TinyMCE
5. **图片存储**：本地文件系统（预留云存储接口）

### 时间约束

- **开发周期**：2周（10个工作日）
- **MVP上线**：1周后（核心功能）
- **全功能上线**：2周后

### 资源约束

- **前端开发**：1人
- **后端开发**：1人
- **UI设计**：0.5人（兼职）
- **测试**：0.5人（兼职）

1. 现有数据库Product表已有description字段，需扩展支持长文本
2. 图片存储空间充足（100GB）

### 数据模型设计

#### content_sections 表（商品详情内容分区）
```sql
CREATE TABLE content_sections (
    id INTEGER PRIMARY KEY,
    product_id INTEGER NOT NULL,
    section_type VARCHAR(50),  -- 类型: story, nutrition, ingredients, process, tips
    title VARCHAR(200),
    content TEXT,  -- 富文本内容
    display_order INTEGER,
    created_at DATETIME,
    updated_at DATETIME,
    FOREIGN KEY (product_id) REFERENCES products(id)
);
```

#### nutrition_facts 表（营养成分表）
```sql
CREATE TABLE nutrition_facts (
    id INTEGER PRIMARY KEY,
    product_id INTEGER NOT NULL,
    serving_size VARCHAR(50),  -- 每份大小
    calories REAL,  -- 热量(kcal)
    protein REAL,  -- 蛋白质(g)
    fat REAL,  -- 脂肪(g)
    carbohydrates REAL,  -- 碳水(g)
    sodium REAL,  -- 钠(mg)
    dietary_fiber REAL,  -- 膳食纤维(g)
    sugars REAL,  -- 糖(g)
    created_at DATETIME,
    FOREIGN KEY (product_id) REFERENCES products(id)
);
```

### UI/UX设计要点

1. **信息架构**：分层展示，避免信息过载
2. **视觉层次**：使用字体大小、颜色、间距区分重要性
3. **交互设计**：支持快速跳转到各个内容分区
4. **图片展示**：大图为主，支持手势缩放和滑动
5. **加载体验**：骨架屏、进度条，减少等待焦虑
